# Complexity-Reduction Plan

> Working document. Delete once executed.
>
> Produced by an audit of all 8 subsystems: 74 reduction candidates found, 59 confirmed
> by adversarial re-verification against actual call sites. Refuted candidates are recorded
> in "Not worth doing" / "Considered and rejected" so they are not re-proposed.


## 1. Verdict

The complexity is not in the hard parts. `internal/streamBridge.ts` (315 lines) and the pull-serialisation chain encode real bugs that Effect's own AsyncIterable bridge does not handle, and they are pinned precisely at `packages/effect-grpc/test/streamBridge.test.ts:120-247`. The excess sits in three structural places: **(a) a whole parallel front-end** — `packages/codegen` (921 lines, 17 files) reimplements every stage of `buf generate`, is used by zero examples, zero tests and zero smoke checks, and `docs/contributors/architecture.md:3` already says the repo has *two* public packages; **(b) information written down four times** — the reflection/health services carry a vendored descriptor, a full protobuf-es TS mirror, a hand-written Effect `Schema` mirror and a four-direction converter ladder, of which `GrpcReflection.ts:314,318` and `GrpcHealth.ts:120,135` read exactly three symbols (`ServerReflection` ×2, `Health`) out of 838 lines; and **(c) fan-out by call kind instead of by shape** — the same four-way switch is spelled five times in `GrpcServerProtocol.ts` and `GrpcMethodRegistry.ts` when the real axis is binary, and the test suite then hand-rolls the same `captureImplementation`/`handlerContext`/four-entry fixture harness in five files with no shared support module.

Root causes, honestly: **a second front-end nobody uses**, **hand-written codegen output that diverged from the generator it should have come from**, and **no test-support layer**, which turned every new scenario into a fresh 40-90 line harness.

Headline: **~5,900 lines removable of 22,814 tracked (excl. lockfile/snapshots/LICENSE) — 26%.** Of that, **~2,800 of 7,749 published-source lines (36%)** and **~1,400 of 9,303 test lines (15%)**. No load-bearing behaviour is lost; the only user-visible cuts are `GrpcAuth`, `tracestate` propagation, the v1alpha reflection alias, and four inert generator options.

## 3. Phases

**Phase 1 — Delete what nothing references.** Goal: remove entire files and packages with zero call sites, before any refactor has to account for them.
Moves **1, 2, 5, 8**, plus the dead-code half of **4** (`detectImportCycles`, `naming.ts` orphans, `packageName`, `wellKnownFields`). **−2,589.**
First because every one of these is a pure deletion whose only verification is "the build still passes" — and because move 1 removes 921 lines that moves 12 and 13 would otherwise have to be threaded through (`packages/codegen/src/options.ts` mirrors the plugin's option parser byte-for-byte).

**Phase 2 — Collapse the test suite.** Goal: one harness per primitive, one test per behaviour.
Moves **7 → 3 → 10**, plus the plugin-fixture half of **4**. **−1,194.**
Strict order: delete the middle protocol-harness layer *first* (move 7 removes `captureImplementation`/`handlerContext` from `health.test.ts` and `reflection.test.ts` outright), then extract `test/support/serverHarness.ts` from the two or three files that survive, then re-point telemetry at it. Doing 3 before 7 means building a shared harness and immediately deleting half its consumers. Phase 2 must precede Phase 4 so the runtime refactors land against a suite that already has one assertion per behaviour rather than three.

**Phase 3 — Cut features nobody asked for.** Goal: shrink the public surface from 13 namespaces to 11 before locking anything else in.
Moves **9, 12, 13, 11**. **−1,082.**
These are the breaking changes; group them into one release so consumers (there are none, per `AGENTS.md`) absorb one break. Move 12 must land before any work keyed on `service.methods` — with the `methods` cardinality filter alive, a descriptor-driven helper silently re-adds methods the emitter deliberately dropped (`generate.test.ts:720-737` asserts `UploadUsers` is absent).

**Phase 4 — Refactor what remains.** Goal: remove the structural duplication that would otherwise regrow.
Moves **6, 14**, plus the duplication half of **4** (well-known list, value-model dispatch, `unsupported.ts`). **−1,035.**
Last because these are the only moves that change working code paths rather than deleting them, and because move 6's `handlersEffect` change touches the same `GrpcNodeServer.serveAll` path that move 14's merged example server exercises.

## 5. Structural bets

**Bet 1 — Make the two built-in services generated-shaped, not hand-shaped.** This is move 11(a) taken as a policy rather than a one-off. Today `GrpcHealth` and `GrpcReflection` are hand-written codegen output that drifted: the plugin renders oneofs as `Schema.Struct({case, value})` (`generateSchemas.ts:80-84`) and emits pass-through converters, while `GrpcReflection.ts:47-58` documents "the `message_request` oneof is flattened into optional fields" and pays 172 lines for the translation. Adopting the generated representation for the built-ins means every future built-in service (a `channelz` implementation, say) costs the emitter's line count, not twice it. **Cost:** breaks two exported schema types, and `GrpcHealth`'s `ServingStatusSchema` deliberately uses `Schema.Literals` against a numeric wire enum where the generator emits `Schema.Number` — that divergence is genuine and should *stay* (it is better API), so the alignment is partial. **Benefit:** −120 now, and it removes the precedent that says "the built-ins are special".

**Bet 2 — Route the plugin through protoplugin's `GeneratedFile` instead of string arrays.** `pluginDefinition.ts:57-62` renders a whole file to a string and replays it through `generated.print(line)`, so protoplugin's import machinery is never used. That single decision forces `generate.ts:29-71` (hand-assembled import statements), `naming.ts:6-15,22-32` (hand-computed relative paths and extensions), and four `FileUsage` fields whose only job is deciding whether an import line prints — including the 9-line-documented `usedImportedTypes`, which exists purely because an unreferenced alias trips consumers' `noUnusedLocals`. `GeneratedFile.import(name, from, typeOnly?)` returns an `ImportSymbol` emitted only when actually printed, paths are made relative automatically, and `importSchema(DescService)` imports the protoc-gen-es descriptor directly, replacing `pbImportPath` and the `descriptorImports` list. **Cost:** ~−80 net (ImportSymbol declarations cost lines back), medium risk, and three prerequisites the dossier's own reviewer surfaced: `usesBase64Bytes` must *stay* (it gates the `fromGrpc$Bytes`/`toGrpc$Bytes` helper emission at `generateRegistry.ts:296`, not just an import line); `effectFileName` (`naming.ts:17-20`) must stay (it names the output file at `pluginDefinition.ts:58`); and the empty-model guard at `pluginDefinition.ts:49-56` only becomes redundant if the header moves to `f.preamble(file)`, which replaces the current `// @generated by protoc-gen-effect-grpc` text with protoplugin's format. Verify first that `f.import(name, "./demo/v1/x_effect_grpc.ts")` extension-rewrites paths to the plugin's *own* sibling outputs, not just protoc-gen-es `_pb` files. **Benefit:** the fields that keep regrowing (one new import gate per feature) stop existing.

**Bet 3 — Move per-service registry/client/server glue into runtime helpers keyed on `DescService`.** Every generated registry entry restates `kind`, `tag`, `service` and `localName`, all four already on the `DescMethod` reachable from the `DescService` the entry imports. Add `GrpcMethodRegistry.fromService(service, schemas)` (~30 runtime lines), `GrpcServerProtocol.serviceHandlersLayer(service, implementation)` (delegating to `handlersLayer`, *not* reimplementing `bindHandler` — `GrpcServerProtocol.ts:114-121` merges build-time context beneath per-call context on purpose), and `CodegenSupport.makeClient<Iface>(service)`. The emitter's per-method blocks (`generateRegistry.ts:17-49`, `generateClient.ts:8-32,51-68`, `generateServer.ts:7-28,43-53`) collapse to one line each. **Cost, stated honestly: net source is roughly flat (~−84 emitter, ~+90 runtime, i.e. ≈ +6).** The win is generated output — ~−114 across the two checked-in examples plus proportional snapshot shrink — which compounds per downstream user and per method but is not a code deletion. Two real losses: `generateClient.ts:23` emits `} satisfies ${ClientService};`, a compile-time check *inside* the generated file verified by `tsc` on every regeneration, which becomes an unchecked type parameter; and `generateServer.ts`'s `as <Type>` casts are what keep `handlerBinding` honest against `implementationSignature`. **Sequencing:** land move 12 first, or `service.methods` will silently re-add methods the `methods` filter deliberately dropped. `fromService` must be additive — `GrpcHealth.ts:115-145` and `GrpcReflection.ts:292-320` hand-build registries with non-descriptor-derivable converters. **Verdict:** take this only if you believe downstream generated-file size matters more than repo source size. Given "no users yet", I would defer it past the other 14 moves.

## 6. Not worth doing

- **`handleEffectCall` / `handleStreamCall` (`GrpcServerProtocol.ts:176-238, 247-345`).** These do implement the same span/status/abort protocol twice, but with genuinely different mechanics (`Effect.raceFirst` + `Effect.withSpan` vs a scoped span with a signal listener closing the pump). Expressing the Effect path on top of the stream path is 65 lines for high risk on the single hottest code path in the library. The ratio is the worst in the dossier. Leave it; revisit only if a bug forces you into that code anyway.
- **`internal/streamBridge.ts` (315 lines).** `ownedPull`, `detachedCleanup` and the pull-serialisation chain each encode a specific bug Effect's own AsyncIterable bridge does not handle (abandoned pulls on `return()`, a queued `return()` deadlocking a mid-pull teardown, overlapping pulls duplicating chunk elements). `test/streamBridge.test.ts:120-247` pins exactly those. This is the code the library exists to provide.
- **`GrpcInvoker` as a seam.** It looks like a wrapper, but `generateClient.ts:61-67` emits only `invoker.{unary,serverStream,clientStream,bidiStream}` — every generated client depends on it and on nothing else. Keep it; just make sure it stays *one* implementation.
- **`generate.test.ts:28-110,113-430` (the hand-built model tests).** They look redundant against the `.proto` fixture harness, but `empty_messages.proto` is not all-empty (it declares `GetStatusResponse { string status = 1; }`), no fixture has a self-referencing message, and no fixture declares a streaming method — so three of those eight tests are the *only* coverage of their branches, including the recursive-edge branch at `generateSchemas.ts:114-116`. Also, `proto-features.test.ts` shells out to `pnpm exec buf build` with a 120 s timeout; deleting the model tests leaves `generate.ts`/`generateSchemas.ts`/`generateRegistry.ts` with zero coverage runnable without the buf toolchain. Take only the ~28-line `toContain` trim in move 4(e).
- **`packages/protoc-gen-effect-grpc/test/__snapshots__/proto-features.test.ts.snap` (2,295 lines).** It is generated data, not hand-maintained code, so it does not carry the maintenance cost the AGENTS.md rule targets. Pruning it trades ~150 lines of generated text per feature for the loss of that feature's only structural emission pin, and two of the fourteen features (`empty_messages`, `bytes_collision`) have no `converts …` test at all. One change *is* worth making, as a standalone commit with no pruning: convert the single `it` to `it.each` with explicit `toMatchSnapshot(featureName)` keys and run `-u` once, so snapshots stop being positionally numbered `1..14` and inserting a feature stops renumbering every subsequent one.
- **`packages/codegen`'s internal refactors** (raw `--opt` passthrough, `BufBuildError` collapse, `index.ts` trim, temp-`buf.gen.yaml` rewrite). All are subsumed by move 1. The passthrough one is also broken as specified: `run.ts:68-69` builds a *different* parameter string for protoc-gen-es (`target=ts,import_extension=<ext>`), which would reject forwarded `methods=`/`errors=` tokens.

### Considered and rejected (instructive)

**Replacing the in-memory invoker with connect's `createRouterTransport`.** Superficially perfect — `GrpcServerProtocol.make` already returns the exact `(router: ConnectRouter) => ConnectRouter` shape that transport wants, and it would have deleted ~620 lines including `internal/inMemoryInvoker.ts` and all of `invokerParity.test.ts`. It fails on the protocol: `createRouterTransport` builds `createTransport` from `protocol-connect/transport.js`, i.e. the **Connect** protocol, not gRPC. The timeout header becomes `Connect-Timeout-Ms` instead of `Grpc-Timeout`, and handlers would additionally see `Connect-Protocol-Version` and `content-type: application/connect+proto` — breaking exactly the invariant `invokerParity.test.ts:41-56` exists to pin, in a package whose README calls itself a native gRPC runtime. Two further blockers: every in-memory test uses fabricated tags with no descriptors (`"test.Svc/Unary"`), and the repo ships no client-streaming or bidi descriptor at all, so those tests could not be expressed without a new `.proto` fixture; and `GrpcServerProtocol.make` runs handlers through `Effect.runPromiseWith` on a detached runtime, which turns `grpcInvoker.test.ts:432-459`'s synchronous `finalized === 1` assertion into an async race. If the goal is the `docs/users/limitations.md:26-28` caveat (in-memory does not enforce `timeoutMs` on stream shapes), fix it directly with ~10 lines of `Stream.timeoutFail` in `inMemoryInvoker`'s `serverStream`/`bidiStream`.

**Replacing emitted per-message `from*`/`to*` converters with a descriptor-driven runtime codec.** The biggest single number in the dossier (claimed −570) and the most tempting. It fails because the four converter fields are not an over-abstraction but a load-bearing extension seam with a real second implementation: `GrpcHealth.ts:115-145` and `GrpcReflection.ts:292-320` hand-build registry entries whose converters *deliberately* diverge from anything a descriptor codec can produce (`ServingStatusSchema` is `Schema.Literals` against a numeric enum; the reflection oneof is flattened). Both are exported public API. It also inverts `docs/contributors/architecture.md`'s stated contract ("Runtime code should not need to inspect `.proto` files"), and ~15 round-trip tests call `entry.fromGrpcRequest(...)` directly and would all need rewriting through `encodeRequest`/`decodeRequest`, changing every asserted value. Honest arithmetic: ~511 source lines removed against a runtime codec that realistically costs 250-350 (bytes base64, bigint↔string, Timestamp ISO, Duration `{_tag}`, nine wrappers in *both* bare and boxed encodings, Any, four protobuf-JSON WKTs, oneof's null/undefined asymmetry, number-vs-string map keys, `compact()`) — roughly −180 net source, most of the headline being generated artifacts. If you ever want it, the only viable shape is making the four entry fields *optional* with the descriptor codec as the default, which preserves the seam but adds runtime code rather than removing it.
